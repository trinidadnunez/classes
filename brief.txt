Product Overview
Exercise Class Selector is a web-first marketplace that aggregates exercise classes across Buenos Aires (yoga, HIIT, Pilates, etc.), enabling residents to discover nearby studios and reserve classes directly from the platform. It targets Buenos Aires residents who want a simple way to explore class options across multiple studios and book seats without visiting multiple websites. The core problem it solves is fragmented discovery and friction around reserving classes across independent studios by centralizing listings, schedules, and the reservation flow into one searchable, location-aware web experience.

Core User Flows
1) Discover & Browse Classes: (a) User opens home page, sees search bar + location context (Buenos Aires by default). (b) User searches by class type, studio, or address or browses categories and map. (c) System returns list + map with cards showing next class time, price (if provided), capacity and free spots. (d) User clicks a class card to open Class Details.

2) View Class Details & Reserve: (a) On Class Details, user sees schedule, instructor, capacity, exact time, images, studio location, transport info, and rules. (b) User clicks 'Reserve' -> sign-in prompt for Guest, or direct booking for signed-in User. (c) User selects date/time instance (if recurring) and confirms booking. (d) System creates provisional booking, decrements available spots, shows confirmation UI and sends email/SMS confirmation (if enabled). (e) User may add to calendar (ICS) and receive reminders.

3) Manage Bookings & Profile: (a) User navigates to My Bookings, sees upcoming and past bookings. (b) User can cancel (per studio policy), join waitlist if class full, or reschedule where enabled. (c) System enforces cancellation windows and updates availability; sends cancellation notifications; exposes booking history and ability to leave reviews later.

4) Studio Onboarding & Class Publishing (Studio Manager flow): (a) Studio signs up, fills profile, address, facilities, instructor list, pricing model. (b) Studio creates classes/schedules (single or recurring), sets capacity, booking rules, cancellation policy. (c) Studio publishes class; system validates and lists it for discovery. (d) Studio dashboard shows bookings, attendee list, waitlist and basic analytics.

5) Admin / Operations: (a) Admin reviews studio signups, can approve/decline listings, manage content moderation, and handle dispute resolution. (b) Admin dashboard shows health metrics, studio compliance issues, and system alerts.

Ux Ui Design System Patterns
Design system: responsive web-first UI optimized for desktop and mobile browsers; base components: header with persistent search, left/overlay filter panel, map and list dual view, card pattern for classes, studio profile pages, and a compact mobile bottom-sheet for class details. Core patterns: global top nav (logo, search, sign-in, my bookings), card grid/list with consistent microcopy (duration, capacity, next time), progressive disclosure (expandable class details), calendar picker with timezone normalization, modals for booking flows (single focus, clear CTAs), inline validation on forms, toast notifications for success, accessible alerts for errors. Accessibility: WCAG 2.1 AA target, keyboard navigable filters and booking steps, alt text for images. Visual style: neutral, energetic palette, clear CTAs (primary accent for Reserve), legible sans-serif typography, 8pt grid spacing. Design references: Material Design for spacing and components, Airbnb experiences listing patterns for discovery and maps, ClassPass for marketplace behavior. Interaction rules: minimize modals, prefer in-page progressive panels for booking steps, use skeleton loaders for async lists, and confirm destructive actions with explicit confirmation dialogs.

Functional Requirements Phased
MVP (must-have): Search & discovery (text + category + location), studio and class listings, class detail pages, user accounts (signup/login via email), reservation system without payment (seat allocation, confirmation emails, calendar export), studio dashboard for basic class publishing and capacity management, basic notifications (email), admin portal for studio approvals and moderation, maps integration for studio locations, responsive UI. V1.1 (post-launch priority): Payment integration (MercadoPago + Stripe fallback) for paid classes, calendar sync (Google/Apple), SMS notifications, waitlist functionality, review & rating system, basic studio analytics (bookings, capacity utilization), coupon/promo codes, improved onboarding flows and verification (ID/phone). V2+ (future roadmap): Mobile native apps (iOS/Android), subscription/pass products, dynamic pricing, multi-studio bundles, deeper analytics & partner integrations, marketplace partnerships, corporate/group booking flows, advanced CRM capabilities for studios, integrations with third-party studio management systems (Mindbody, WellnessLiving), and localized multi-language support (Spanish primary, optionally English).

Feature Specifications
Search & Filters: What it does: returns classes matching text, category, date, location radius, price, time of day, difficulty level and instructor. UI: top search bar, left filter rail, map sync; autosuggest for studios and classes. Acceptance: search returns results within 2s, filters combinable, empty-state guidance. Use cases: find 'Pilates near Palermo tomorrow morning'.

Class Detail Page: What it does: shows class metadata, schedule instances, instructor, images, studio location, rules, capacity and price (if present). UI: hero with class title, schedule selector, Reserve button, map and directions, CTA to add to calendar. Acceptance: schedule selector shows available instances, capacity shown, CTA disabled if full.

Reservation Flow: What it does: reserves a spot, confirms by email, updates availability. UI: modal or in-page panel; for guests show sign-in/signup before finalizing. Acceptance: booking completes transaction without error; available spots decrease atomically; confirmation sent within 1 minute. Use cases: signed-in user reserves single session; guest signs up and reserves.

Booking Management: What it does: list of upcoming/past bookings, cancel/reschedule when allowed, join waitlist. UI: My Bookings page with stateful actions and calendar export. Acceptance: cancellation follows studio policy, waitlist notified automatically when seat frees.

Studio Dashboard / Onboarding: What it does: studios create profile, publish classes, set capacity, manage attendees and schedules. UI: multi-step onboarding, class editor, attendee list with check-in export (CSV). Acceptance: studios can publish a class and it appears in search within 15 minutes.

Notifications: What it does: sends booking confirmations, reminders, cancellations, waitlist promotions. UI: email templates, optional SMS. Acceptance: scheduled reminders sent 24h/2h prior; failure logged for retry.

Admin Tools: What it does: review/approve studios, content moderation, system health. UI: admin portal with filters, manual overrides for bookings. Acceptance: admin can deactivate studio and notify affected users.

Future Payment Integration: What it does: capture payments, issue refunds, handle partial failures. UI: payment page in booking flow; store payment methods securely. Acceptance: successful captured payments and receipts issued; payment failures surfaced with retry guidance.

User Roles Permissions
Guest: browse listings, use search and filters, view class details, initiated reservation but must sign-up to complete booking. Registered User: everything Guest can do plus complete reservations, manage bookings, add to calendar, write reviews, save favorite studios/classes, receive notifications and view booking history. Studio Manager (Studio Account): create and manage studio profile, publish/edit classes/schedules, set capacity and rules, view attendee lists, export check-in lists, moderate their content, view basic analytics. Platform Admin: approve/decline studio signups, moderate content and reviews, manage users and studios, override bookings, access platform-wide analytics, manage support tickets and system configuration. Note: Only Admins can change critical platform settings; Studio Managers cannot change pricing policies globally. Role restrictions: Studio Managers limited to their own studio scope; Registered Users cannot change studio content; Guests cannot complete bookings.

Data Model Key Objects
User: id, name, email, phone, password_hash, profile_picture, preferences, joined_date, last_active. Studio: id, name, address, geo_coordinates, description, images, facilities, operating_hours, contact_info, manager_user_id, status(approved/pending/disabled). Class (Offering): id, studio_id, title, description, category, difficulty, duration, price(optional), instructor_id, recurring_rules, default_capacity, tags, images. Session (Scheduled Instance): id, class_id, start_time, end_time, capacity_override, spots_remaining, status(scheduled/cancelled). Booking: id, user_id, session_id, status(confirmed/cancelled/waitlisted), created_at, cancel_deadline, checkin_status. Instructor: id, name, bio, certifications, contact. Review: id, user_id, class_id/studio_id, rating, comment, created_at. PaymentTransaction (future): id, booking_id, amount, currency, provider, status, payment_method_id, created_at. PromoCode: code, studio_id(optional), discount_type, value, validity_start, expiry, usage_limit. Relationships: Studio 1->N Classes; Class 1->N Sessions; Session 1->N Bookings; User 1->N Bookings; Instructor 1->N Classes; Booking optionally links to PaymentTransaction.

Business Rules Logic
Capacity & Booking: bookings decrement spots_remaining atomically; sessions cannot be overbooked beyond capacity. Waitlist: when session full, users may join waitlist; when a spot frees, notify and auto-promote first waitlist user if within promotion window. Cancellation: studio-defined cancellation window (e.g., 24 hours default); cancellations within window allowed and free; outside window may be blocked or flagged (payments/refunds logic added when payments enabled). Double-booking: prevent a user from booking two sessions with overlapping times (configurable). Studio Approval: studios must be approved by Admin before classes appear in search. Visibility Rules: only approved, active sessions with status 'scheduled' and start_time in future are shown. Booking Confirmation: booking is confirmed once created and stored; for paid classes, booking confirmed only after successful payment capture. Timezone & Date Logic: all times stored in UTC and localized to America/Argentina/Buenos_Aires (UTC-3) for display, honoring daylight rules. Refunds & Charges (future): refunds follow studio's policy; platform takes configurable commission; payments processed through MercadoPago or Stripe with provider-specific failure handling. Data Retention: user personal data retained per local regulation and privacy policy; user can request deletion following platform procedures.

Edge Cases Error Handling
Concurrent Bookings (race conditions): when two users try to book the last spot, ensure server-side atomic decrement and return a clear message to the losing user 'Sorry, the class just sold out; join the waitlist?'. Session Cancelled by Studio: notify all booked users immediately with options to rebook or request refund (if paid). Network / Partial Failure During Booking: if booking creation succeeds but email fails, show success to user and queue notification retry; log and surface to admin if retries fail. Payment Failure (future): if payment fails, leave provisional booking for short hold then release seat; notify user to retry. Double-Booking Attempt: prevent and explain conflict with option to cancel other booking. Invalid Schedule Entry by Studio: on class publish validation, reject overlapping sessions for same studio room/time and provide clear validation errors. Capacity Override Errors: if studio edits capacity lower than existing confirmed bookings, block or require manual resolution by Admin/studio with notifications to affected users. Rate Limit / Abuse: implement API rate limiting; when triggered show generic 'too many requests' error and guidance. Map/Geo Failures: fallback to textual address if map provider fails. Notification Delivery Failures: retry with exponential backoff; surface critical failures in admin dashboard. Missing Data: show graceful placeholders and contact studio link.

Success Metrics Analytics
Core events: search_performed, filter_applied, listing_viewed, class_detail_viewed, initiate_booking, booking_confirmed, booking_cancelled, waitlist_joined, waitlist_promoted, studio_signed_up, studio_approved, studio_published_class, user_signed_up, login, review_submitted. Conversion funnel metrics: % sessions from search -> detail -> initiate_booking -> booking_confirmed. Activation/engagement: DAU, MAU, 7/30-day retention, average bookings per active user, repeat bookers rate. Business metrics: number_of_active_studios, bookings_per_studio, average_capacity_utilization, revenue (post-payment launch), average_order_value, refund_rate, cancellation_rate, average_time_to_confirm. Marketing metrics: CAC (customer acquisition cost), LTV (lifetime value). Quality metrics: booking_failure_rate, average_response_time for search, page_load_time, NPS and app ratings. Tracking implementation: use analytics (GA4 + product analytics like Mixpanel/Amplitude) and event schema with user_id and studio_id where available, and attribute sessions to acquisition channels. Alerting thresholds: booking_failure_rate >1% or page load >3s triggers pager/ops notification.

Technical Constraints Dependencies
Platform: web-first responsive SPA (React or similar) with server-side APIs (Node/Express or Python/Django). Database: relational DB (Postgres) for schedules and bookings; use transactions for atomic seat allocation. Map provider: Google Maps (billing/account) or OpenStreetMap + Leaflet as cost-conscious alternative. Email provider: SendGrid or Mailgun; SMS provider: Twilio (optional, pay-per-message). Payments: MercadoPago primary for Argentina; Stripe as fallback for international payments. Third-party integrations: calendar ICS export and optional Google Calendar API, potential future integrations with studio management systems (Mindbody). Hosting: cloud provider (AWS/GCP) with autoscaling; CDN for static assets. Performance: page load <=2.5s on 3G simulated mobile, booking API latency <500ms under normal load. Concurrency: booking endpoint must handle concurrent reservations and ensure ACID behavior. Data residency and compliance: store PII securely, comply with Argentina data protection expectations; TLS everywhere. Offline constraints: map and schedule availability relies on studio data freshness; storefront must handle stale data for a limited window and reconcile on read. Dependencies: studio cooperation for data correctness, payment provider account approvals, mapping API keys, email/SMS accounts. Cost constraints: start with minimal third-party billings then expand; prefer free tier-friendly providers for MVP (OpenStreetMap, SendGrid free tier).

Testing Qa Requirements
Unit tests: for core booking logic, capacity handling, date/time conversions, and business rules. Integration tests: end-to-end reservation flow (signup -> booking -> confirmation), studio onboarding -> publish -> appears in search. E2E tests: core user journeys (discover -> book), booking race condition simulation, waitlist promotions. Performance/load tests: booking endpoint under concurrent requests simulating peak load (e.g., 200 concurrent bookings on a popular session) and page load tests for catalog pages. Security tests: auth flows, permission enforcement, injection and XSS checks, secure storage of PII. Cross-browser & responsive tests: Chrome, Firefox, Safari, Edge and mobile breakpoints. Accessibility tests: automated a11y checks + manual keyboard navigation and screen reader spot checks targeting WCAG 2.1 AA. Acceptance criteria: all critical flows pass E2E with 0 reproducible failures; booking race tests must show zero overbooking across repeated runs; performance targets met (booking API <500ms median, page load under 2.5s median). Regression tests: run on every release, with test coverage thresholds for critical modules. User acceptance: QA sign-off on smoke tests and staged pilot with 10-20 studios and 100+ users before public launch.

Definition Of Done
MVP launch is complete when: core features implemented and validated (search, listings, class detail, reservation without payment, user accounts, studio onboarding and basic dashboard, admin moderation), automated unit and integration tests passing, E2E tests for primary journeys passing, load tests meeting performance targets, basic accessibility compliance (WCAG 2.1 AA checks completed), email notifications working and verified, maps and geolocation functioning, analytics events instrumented for core metrics, monitoring and error reporting configured (Sentry/CloudWatch), legal/privacy pages and terms published, support flows defined (help center and contact email), deployment pipeline configured and production deployment completed, at least one pilot group of studios onboarded and test bookings completed, and product owner + engineering + QA + operations sign-off. Production-ready means stable behavior under expected load, observability in place, documented runbook for common incidents, and a go-to-market plan agreed. Final sign-off criteria: executive product owner approval, ops readiness checklist complete, and pilot KPI baseline met (e.g., successful bookings >100 and capacity utilization demonstrating platform value).
